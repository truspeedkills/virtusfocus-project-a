DEV PARTNER TECHNICAL HANDOFF — HANDOFF PROMPT
==============================================
SESSION OBJECTIVE:
Produce a comprehensive technical handoff document for the
VirtusFocus app development team. This document defines
EVERYTHING the backend team must build to support the AI
coaching pipeline — data contracts, deterministic computations,
API boundaries, database schema requirements, timing rules,
and rendering specifications.

The audience is a development team that has NO context on the
AI pipeline internals. They need to know: what data to
collect, what to compute, what to send to the AI pipeline,
what to receive back, and what to render to end users.

OUTPUT FORMAT: Professional Word document (.docx) with clear
sections, tables, and code-formatted schemas. The document
should be self-contained — a developer should be able to
implement the backend from this document alone without reading
any pipeline specification files.
==============================================
CONTEXT — WHY THIS DOCUMENT EXISTS
==============================================
VirtusFocus has a fully specified 7-stage AI coaching pipeline
that transforms athlete behavioral data into coaching output
for athletes, coaches, and parents. The pipeline specifications
are written for Claude (the AI agent operating each stage).
The app development team needs a DIFFERENT document — one that
describes only what the backend must implement, with clear
boundaries between "your code" and "the AI handles this."

KEY PRINCIPLE: The app backend handles all DETERMINISTIC
computations (data extraction, formula-based scoring, rule-
based signal determination). The AI pipeline handles all
GENERATIVE work (narrative coaching messages, analytical
reports, contextual interpretation). The backend never calls
an LLM for deterministic work. The AI never receives raw app
event data — only structured JSON intermediaries.

Source: VF_PPD_Scoring_Logic.txt

SECTION 4: ABI SCORING (BACKEND COMPUTATION)
Source: VF_ABI_Scoring_Logic.txt
- Formula: pillar_score = round((Q_a + Q_b) / 2 * 2)
- 4 pillars with input question mapping:
  Ownership (Q7, Q8), Composure (Q9, Q10),
  Focus (Q11, Q12), Structure (Q13, Q14)
- All 25 valid input combinations per pillar with outputs

- Individual pillar bands: Low 2-4 / Moderate 5-7 / High 8-10
- Primary Emphasis rules (within 2 of lowest, max 2, priority

- Output schema: abi_scores object + abi_raw_inputs object

SECTION 5: DAILY DATA COLLECTION & MINI-JSON
Source: VF_Daily_Mini_JSON_Specification.txt,
       VF_App_Input_Format_Specification.txt
- What the app collects daily:
  Morning Tune-Up (focus word, quick win items, mindset
    challenge acceptance, timestamp)
  Evening Review (WTD 5 questions, journaling 3 domains,
    Bullseye 3 rings as arrays, mindset challenge follow-
    through, component sequence, timestamp)
- 4-category evening timing model:
  on-time (before 10 PM), late (10 PM to backfill boundary),
  backfill (2 hours before hard-out), missed (after hard-out
  lockout). Hard-out = Morning Tune-Up release time (default
  6 AM, configurable per program).
- Partial data preservation on hard-out lockout
- Daily Mini-JSON schema: the structured intermediate that
  the backend assembles from raw daily events. This is what
  Stage 6 (Daily Coaching Engine) and Stage 7 (Daily Coach
  Signals) consume.
- Mini-JSON generation timing: after hard-out lockout,
  before Morning Tune-Up release
- Tiered operation: Base = 7 Mini-JSONs/week (Mon-Sun),
  Premium = 7 Mini-JSONs/week but Stage 6 only generates
  snippets Tue-Sun (6/week)
- Deliberately excluded fields (compliance — never in
  Mini-JSON): trigger_method, reminder timestamps, session
  duration, login counts, screen time

SECTION 6: WEEKLY DATA ASSEMBLY
Source: VF_App_Input_Format_Specification.txt
- Weekly Input Object: 7 daily event records + 1 weekly
  check-in record
- Weekly check-in contents:
  Motivation Inventory (5 questions),
  self_ratings (confidence_level 1-10, habit_consistency_level
    1-10),
  season_context (current_season_phase: Off-Season /
    Pre-Season / In-Season / Post-Season / Transition),
  forward_anchor (forward_anchor_text: single-line string,
    submitted_at timestamp)
- Assembly timing: after 7th daily event is finalized (Sunday
  hard-out), before Stage 2 (Interpretation Engine) runs
- The Weekly Input Object is what Stage 2 consumes for App
  athletes. Core Foundation athletes submit a PDF recap
  instead (different input path).

SECTION 7: EXECUTION SIGNAL COMPUTATION (BACKEND)
Source: VF_Execution_Signal_Composite_Score_Rules.txt,
       VF_Coach_Flags_Specification.txt
- 7 composite scores — ALL computed by backend, not AI:
  1. Ownership Index (3 components, 50/30/20 weights)
  2. Drift Score (5 components, 25/20/20/20/15 weights)
  3. Follow-Through Score (2 components, 70/30 weights)
  4. Rhythm Score (4 components, 25/25/30/20 weights)
  5. Review Quality Score (3 components, 40/35/25 weights)
  6. Recovery Score (3 components, 60/25/15 weights,
     event-triggered)
  7. Reactivity Risk Score (4 components, 30/25/20/25 weights)
- For each score: exact formula with component definitions,
  input field mappings, band thresholds, edge cases
- Output: { score: integer 0-100, band: categorical enum }
- 15 coach flags — ALL evaluated by backend:
  Flag object: { flag_id, severity, label, trigger_source,
    description }
  3 severity tiers: monitor (4 flags), attention (8 flags),
    action (3 flags)
  4 categories: score-band threshold, cross-score convergence,
    raw metric pattern, cross-week escalation
- For each flag: exact trigger condition, description template
  with dynamic variables
- Cross-week data requirements: flags 13-15 require prior
  week data. Recovery Score requires prior week for cross-week
  recovery. Streaks require all prior week data.
- Self-ratings alignment classification:
  confidence vs Ownership+Follow-Through → Aligned/Conflated/
    Undervalued
  habit consistency vs Rhythm Score → Aligned/Conflated/
    Undervalued
- Core Foundation fallback: all scores = insufficient data,
  flags = empty array

SECTION 8: DAILY COACH SIGNAL (STAGE 7 — BACKEND)
Source: VF_Daily_Coach_Signal_Determination_Rules_v1.0.txt
- Stage 7 is ENTIRELY deterministic — the backend implements
  this, no LLM involved. Zero AI cost.
- Input: Daily Mini-JSON + optional weekly calibration cache
- Output: Traffic Light (Green/Yellow/Orange/Gray) + 4
  contributing signals + 2 hidden internal inputs
- Contributing signals (visible to coaches):
  Day Outcome: Aligned / Building / Reset Day
  Focus Alignment: Centered / Mixed / Drifting
  Follow-Through: Demonstrated / Gap
  Weekly Trajectory: Building Week / Steady Week / Cooling Week
- Hidden inputs (internal scoring only, never displayed):
  Reflection Breadth, Morning Engagement
- Traffic Light determination: 15-step computation sequence
  (Gray check → point scoring → standard thresholds →
  calibration modifiers → final assignment)
- Point scoring system: -4 to +6 range from 5 signal inputs
- Weekly calibration modifiers from Interpretation JSON:
  emotional_intensity High → +1 shift first 3 days,
  growth_phase adjustments
- Calibration cache: per-athlete key-value store, updated
  when new weekly Interpretation JSON arrives, EI modifier
  self-expires via day check
- Team/position group 4-tier aggregation:
  Strong Alignment / Positive Trend / Mixed Signals /
  Needs Attention (percentage-based thresholds)
- 12 edge cases with handling rules
- Parent signal label mapping:
  Green → Good Day, Yellow → Building Day,
  Orange → Reset Day, Gray → No Update Today
  Contributing signals HIDDEN from parents.
  Team aggregation NOT shown to parents.
- Timing: computed between Mini-JSON generation and Morning
  Tune-Up release

SECTION 9: AI PIPELINE INTERFACE
This section defines the API boundary between the backend
and the AI pipeline. The backend does NOT need to understand
pipeline internals — it sends structured input and receives
structured output.

INPUTS TO AI PIPELINE (backend sends):
- Weekly Input Object (Section 6) → Stage 2 (weekly)
- Daily Mini-JSON (Section 5) → Stage 6 (daily)
- Daily Mini-JSON (Section 5) → Stage 7 (daily, but Stage 7
  is backend-computed per Section 8 — only listed here if
  Stage 7 is implemented as a separate service)
- Athlete Snapshot (one-time, after intake) → Stage 1

OUTPUTS FROM AI PIPELINE (backend receives and stores):
- Interpretation JSON (Stage 2) — weekly, per athlete.
  ~2000-4000 tokens. Contains all classified signals,
  composite scores, coach flags, narrative arc, etc.
  Backend stores this and serves calibration data to Stage 7.
- Coaching Message (Stage 3) — weekly, per athlete.
  Plain text, 250-450 words. Athlete-facing.
- Deep Dive (Stage 3) — weekly, per athlete.
  Plain text, 850-1200 words. Athlete-facing.
- Parent Coaching Message (Stage 3) — weekly, per athlete.
  Plain text, 150-250 words. Parent-facing. D2C Premium only.
  Generated only when parent_inclusion = true.
- Daily Coaching Snippet (Stage 6) — daily, per athlete.
  Plain text, 2-3 sentences. Athlete-facing.
- Weekly Performance Insight (Stage 4) — weekly, per athlete.
  Structured text, 10 sections. Coach-facing.
- Parent Weekly Insight (Stage 4) — weekly, per athlete.
  Structured text, 9 sections scan-first format (indicator +
  blurb per section). Parent-facing. D2C Premium only.
- Weekly Team Snapshot (Stage 4) — weekly, per team.
  Structured text, 7 sections. Coach-facing. Requires 5+
  athletes.
- Audit Summary (Stage 5) — weekly, per athlete.
  Internal QA document. Not displayed to any end user.

SCHEDULING:
- Daily cycle: Hard-out → Mini-JSON assembly → Stage 7
  computation → Stage 6 generation → Morning Tune-Up release
  with snippet
- Weekly cycle: Sunday hard-out → Weekly Input Object assembly
  → Stage 2 → Stage 3 → Stage 5 audit → Stage 4 → delivery
- Stage 3 generates 2 or 3 outputs per run (CM + DD + optional
  Parent CM)
- Stage 4 generates 2 or 3 outputs per run (Coach Insight +
  optional Parent Insight + optional Team Snapshot)

SECTION 11: DATA STORAGE & RETENTION
- What needs persistent storage:
  Athlete intake form responses (immutable after submission)
  PPD scores and ABI scores (computed once at intake)
  Daily event data (raw app interactions)
  Daily Mini-JSONs (assembled daily)
  Weekly Input Objects (assembled weekly)
  All AI pipeline outputs (Interpretation JSONs, CMs, DDs,
    Coach Insights, Parent output, Team Snapshots, Snippets)
  Calibration cache for Stage 7 (per-athlete, updated weekly)
  Cross-week data for composite scores and coach flags

SECTION 12: COMPLIANCE REQUIREMENTS FOR BACKEND
- Execution timing data (completion rates, timestamps, login
  frequency, session duration, reminder dependency counts) is
  NEVER exposed to coaches, parents, or athletes in any form.
  Backend computes these internally; only the derived
  categorical signals (composite score bands, traffic lights,
  coach flags) are surfaced through the AI pipeline's
  compliance-safe translations.
- Raw journal text is athlete-private. Coaches see
  compliance-safe summaries only (via AI pipeline output).
- The backend must enforce the hard-out lockout — no evening
  review submissions after hard-out time.
- Parent data boundary: parents see the SAME data as coaches,
  reframed in parent-friendly language by the AI pipeline.
  Backend does not compute separate parent signals (except
  parent label mapping in Stage 7 per Section 8).

SECTION 13: CORE FOUNDATION COMPATIBILITY
- Core Foundation athletes use PDF weekly recaps instead of
  daily app interactions
- Backend detects input mode (App vs CF) and routes accordingly
- CF athletes: no Mini-JSON, no composite scores, no coach
  flags, no daily snippets, no daily signals
- CF athletes still receive: weekly Interpretation JSON,
  Coaching Message, Deep Dive, Coach Insight (all via AI
  pipeline processing the PDF recap)
- CF is transitional — will phase out as athletes migrate
  to App

CRITICAL RULES FOR THE HANDOFF DOCUMENT
==============================================
1. SELF-CONTAINED: A developer reading ONLY this document
   should be able to implement the entire backend. Do not
   say "see VF_Execution_Signal_Composite_Score_Rules.txt
   for details" — include the details.

2. COMPLETE FORMULAS: Every deterministic computation must
   include the full formula with all components, weights,
   thresholds, and edge cases. No summaries of formulas.

3. CLEAR BOUNDARIES: For every piece of functionality,
   explicitly state whether the BACKEND or the AI PIPELINE
   handles it. Use consistent labels: [BACKEND], [AI PIPELINE],
   [BACKEND → AI PIPELINE], [AI PIPELINE → BACKEND].

4. SCHEMA-FIRST: Lead every section with the exact JSON
   schema (field names, types, enums), then explain the
   business logic.

5. COMPLIANCE FORWARD: Every section that touches data
   surfaced to users must note what is PROHIBITED from
   display. The dev team needs to know what NOT to expose
   as much as what to expose.

6. TIER-AWARE: Note which components activate for which
   product tiers. The dev team needs conditional logic.

7. TIMING-PRECISE: Every computation must include when it
   runs, what triggers it, and what depends on its output.
   The dev team builds the scheduling infrastructure.

8. NO AI PIPELINE INTERNALS: The dev team does not need to
   know about prompt engineering, editorial audit criteria,
   coaching voice calibration, or section-by-section content
   guidance. They need data contracts and scheduling.
   Exception: Stage 7 (Daily Coach Signal) IS a backend
   computation — include full determination logic.

9. EDGE CASES: Include all edge cases from the source specs.
   The dev team will encounter them in production.

10. SAMPLE DATA: Include at least one worked example for each
    major computation (PPD, ABI, composite scores, Stage 7
    signal) using real athlete data from the sample files.
