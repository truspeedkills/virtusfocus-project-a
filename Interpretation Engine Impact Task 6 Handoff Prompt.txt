WHAT TASK 6 MUST DECIDE
============================================================

For each group of new snapshot fields (A-F), answer THREE questions:

1. DOES STAGE 2 NEED THIS FIELD?
   - Does this field inform any weekly classification, scoring, or signal?
   - Or is it pass-through data that downstream stages need?
   - Or is it stored for future use only (e.g., longitudinal PPD Alignment Signal)?

2. HOW SHOULD IT ENTER THE WEEKLY JSON?
   Three options per field:
   (a) ACTIVE USE — Stage 2 reads the field and uses it in a classification rule or computation. It gets its own JSON field with a derivation rule.
   (b) PASS-THROUGH — Stage 2 reads the field from the snapshot and copies it into the weekly JSON unchanged, so downstream stages can consume it. No classification rule needed.
   (c) NO ACTION — The field stays in the snapshot only. Stage 2 does not read it. Downstream stages that need it read the snapshot directly (architecture change) or wait for a future enhancement.

3. WHAT CHANGES ARE NEEDED?
   - JSON schema additions (new fields in the weekly output)
   - JSON Rules additions (new classification sections)
   - Project Instructions additions (new processing steps)
   - Downstream routing implications (what Stage 3/4/5 can now do)

============================================================
KEY DESIGN CONSTRAINTS
============================================================

1. DUAL-MODE MANDATORY: Any schema additions must have Core Foundation fallback values. Same pattern as execution_behavior_signals: App athletes get enriched data, CF athletes get neutral/null/"not available" values. Zero degradation.

2. NO DEGRADATION TO EXISTING FIELDS: The 5 baseline_profile fields are locked. All existing classification rules (WTD, Bullseye, RQS, consistency, emotional intensity, narrative arc, growth phase progression, execution behavior signals) remain unchanged.

3. ADDITIVE ONLY: New fields sit alongside existing ones. No existing field is renamed, moved, or retyped.

4. COMPLIANCE: PPD bucket names, ABI pillar scores, and raw intake data NEVER reach coaching staff. Same inform-only/backend-only pattern as execution signals. Stage 4 and Stage 5 compliance rules already prohibit this — but any new JSON fields must be classified correctly (inform-only vs. surface-safe).

5. SCOPE CONTROL: Not everything in the v2.0 snapshot needs to flow through Stage 2. Some fields may be snapshot-only references. The assessment should be honest about what adds value NOW vs. what is future-state.

6. DOWNSTREAM AWARENESS: Stage 3 (v1.5) already references ppd_top_3 coaching implications and ABI primary_emphasis in its field routing rules (VF_Coaching_Output_JSON_to_Message_Map.txt). Stage 4 (v1.2) references ecosystem data for future Parent Output routing. These downstream expectations should inform what Stage 2 passes through.

SPECIFIC INTEGRATION POINTS TO ASSESS
============================================================

These are the highest-value integration questions:

A. ppd_primary_problem + baseline_primary_derailer
   The JSON Rules Section 4C uses baseline_primary_derailer (a prose sentence) for the Developing→Consistent advancement criterion. Should ppd_primary_problem (a bucket name) replace, augment, or sit alongside this check? The bucket name is more deterministic; the sentence is more descriptive.

B. ABI primary_emphasis + coaching personalization
   Stage 3 v1.5 references ABI emphasis for coaching direction. But Stage 3 reads the JSON, not the snapshot. Should primary_emphasis be a pass-through field in the weekly JSON? Or should Stage 2 derive something from it (e.g., a "pillar_focus" field that adapts over time as the athlete develops)?

C. Ecosystem fields + Stage 4 Parent Output
   The Parent Output engine (Idea 4, Tasks 16-21) needs ecosystem data. Should Stage 2 pass these through now, or should the Parent engine read the snapshot directly when it's built?

D. ppd_all_scores + future longitudinal PPD
   ppd_all_scores is explicitly stored "for future longitudinal PPD Alignment Signal." Should Stage 2 carry baseline PPD scores in the weekly JSON now (for eventual comparison with execution-derived PPD), or defer until the longitudinal PPD feature is designed?

E. baseline_profile expansion
   Should baseline_profile in the weekly JSON expand from 5 fields to include PPD/ABI summary data? Or should a separate baseline_intake_data block be created?

F. Season phase + competitive level
   These contextual fields could inform Stage 2's upcoming_context classification or Stage 4's Season Phase Overlay metric. Should they be pass-through fields?
