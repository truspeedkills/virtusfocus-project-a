IDEA 1: ATHLETE INTAKE FORM IMPROVEMENTS
============================================================

Priority: FIRST (foundational — changes to the input layer
constrain everything downstream)

What it is:
-----------
Expand the athlete intake form to capture more structured data
that produces better baseline profiles and ultimately improves
all downstream pipeline output.

CRITICAL CONSTRAINT: This is ADDITIVE, not replacing.
The 4 core baseline fields are LOCKED and do not change:
  - baseline_identity_claim
  - baseline_hinge_habit
  - baseline_primary_derailer
  - baseline_growth_phase

The improvement adds new intake questions that either:
  (a) Create NEW baseline fields alongside the existing 4
  (b) Enrich existing baseline fields with structured data
      that sits alongside the current free-text answers

The key insight: the current intake form relies heavily on
athletes typing written explanations. Many athletes (especially
younger ones) struggle to articulate complex mental performance
concepts in free text. Structured selections (preset options,
scales of 1-10, multiple choice) can capture the same information
more reliably while still allowing free-text where it adds value.

This mirrors the self_ratings_alignment pattern already in the
pipeline — where structured numeric self-assessment (1-10 scales)
gets compared against behavioral execution data to produce
classifications (Aligned/Conflated/Undervalued).

Pipeline touchpoints:
- Stage 1 (Snapshot Generator — new/enriched baseline fields)
- Stage 2 (Interpretation Engine — if new baseline fields feed
  into interpretation logic, alignment checks, or trend analysis)
- Potentially all downstream stages that reference baseline fields
- Same additive design principle as the Execution Signal upgrade:
  new data makes output better without changing what already works

  → Contains: 7-section redesigned form (19 questions), Primary

Key design questions for collaborative discussion:

- How do PPD scores and ABI scores integrate into the existing

IDEA 2: DAILY COACHING SNIPPET
============================================================

Priority: SECOND (architecturally coupled with Idea 3 — both
introduce daily processing)

What it is:
-----------
A short daily coaching message delivered to the athlete each
morning, separate from the Morning Tune-Up but delivered at
similar times. Quick, actionable, personalized.

This is ARCHITECTURALLY NEW. The entire pipeline currently
operates on a weekly batch cycle. This introduces daily output
for the first time.

No detailed specification document exists yet — this is
exploratory. The user wants to investigate:

- Should it be based on yesterday's WTD scores?
- Should it be based on yesterday's journal entries?
- Should it reference the current week's micro-commitment?
- How does it differ from the Morning Tune-Up content?
- What is the right length? (Likely very short — 2-4 sentences)
- Does it need its own pipeline stage, or is it a sub-process
  of Stage 3?
- What data does it consume? (Daily app data, not the weekly
  Interpretation JSON — the JSON doesn't exist yet mid-week)
- How does it maintain Coach Arron's voice at daily scale?
- Does it need audit criteria? (Probably lighter than full
  Stage 5, but clinical language rules still apply)

Pipeline touchpoints:
- New output channel (daily, not weekly)
- Consumes daily app data directly (yesterday's events)
- Must reference baseline snapshot (identity claim, derailer)
- Must maintain non-clinical compliance
- Persona: Coach Arron (same as Stage 3)
- Needs to know the current micro-commitment (from last weekly
  Interpretation JSON) to reinforce it
- Interacts with Morning Tune-Up (must not duplicate it)
- If Idea 1 adds ABI pillar scores, the daily snippet could
  target the athlete's weakest pillar

Key architectural questions:
- Does this require a "daily interpretation" step, or can it
  operate directly on raw daily data + baseline + last weekly JSON?
- How does daily coaching quality stay consistent without the
  full interpretation layer that weekly output benefits from?
- What is the compliance boundary? (Same clinical language rules,
  but does it need the same audit rigor?)
- Can daily snippets be template-driven with variable insertion,
  or do they need full generative output?

IDEA 3: DAILY COACHING INSIGHTS
============================================================

Priority: SECOND (design alongside Idea 2 — shared daily
processing architecture)

What it is:
-----------
Currently all Coach Insights (Stage 4) update weekly. The user
wants to explore having select key analytics update daily for
coaching staff, with the full 10-section Weekly Performance
Insight remaining on the weekly schedule.

No detailed specification document exists yet — this is
exploratory. Questions to investigate:

- Which of the 10 Weekly Performance Insight sections (or which
  metrics) are meaningful at daily granularity?
- What daily data points are useful to coaches WITHOUT violating
  the compliance framework? (NCAA rules still apply — no raw
  execution data, no completion rates, no timing data)
- Is this a subset of existing sections that refresh daily, or
  entirely new daily-specific metrics?
- How do daily insights relate to the weekly full report?
  (Do they roll up? Does the weekly report supersede them?)

Likely candidates for daily refresh:
- Win the Day score (yesterday's 0-5) — simple, already
  compliance-safe
- Day classification (Win/Neutral/Missed) — simple
- Running weekly tier projection — coaches could see mid-week
  trajectory
- Engagement indicator (completed/not completed yesterday) —
  but this borders on compliance territory

Pipeline touchpoints:
- Stage 4 (Coach Insights Engine — currently weekly-only)
- Same compliance constraints as current Coach Insights
  (NCAA-safe, no raw execution data surfaced to coaches)
- Requires daily data consumption — same infrastructure as Idea 2
- The compliance framework (VF_Coach_Insights_Compliance_Framework.txt)
  must be consulted carefully — daily analytics could easily
  cross into participation monitoring

Key architectural questions:
- Does this share the same daily data processing path as Idea 2?
- What is the minimum viable daily insight that is genuinely
  useful to a coach and genuinely compliance-safe?
- Does the daily insight need its own audit criteria?
- How does this interact with the Composite Readiness Signal?
  (Currently weekly — could a daily version exist?)

IDEA 4: PARENT-FACING OUTPUT EXPANSION + PARENT INSIGHTS
============================================================

Priority: LAST (benefits from knowing final intake form +
daily output design)

What it is:
-----------
Two components:

(A) EXPANDED PARENT COACHING MESSAGE
Currently, a lightweight parent-facing message exists alongside
the Coaching Message and Deep Dive. It only includes:
  - growth_phase
  - consistency trend (simplified)
  - one development signal

It does NOT include weekly tier labels, technical classifications,
scoring bands, or execution behavior data.

The user wants to expand this into something more substantial —
real coaching advice for parents on how to support their athlete's
mental performance growth at home. Not to make the parent the
coach, but to give them structured support guidance.

(B) PARENT INSIGHTS DASHBOARD
A dashboard view of insights and indicators similar to Stage 4
Coach Insights, but with a CRITICAL DIFFERENCE: parents are NOT
subject to NCAA compliance restrictions. Therefore parents CAN
see:
  - Direct app usage data
  - Trends and streaks
  - Completion patterns
  - Engagement metrics
  - Participation consistency

This is the INVERSE of the current compliance model — the data
that is most heavily protected from coaching staff (execution
behavior metrics) is exactly what parents are allowed to see.

HOWEVER, one boundary remains absolute: the athlete's private
journaling content and personal written reflections are NEVER
shown to parents. Honest reflection requires psychological
safety — if athletes believe parents see everything they write,
reflection quality drops.

Document to read:
- D:\OneDrive\Documents\(TEST) Project A\Agents - Generators\Parents Insight Engine\D2C Parent App Insights and add ons.txt
  → Contains: Full Parent App overview — purpose, D2C model
    ($5/month add-on), 3 core functions (visibility, context,
    guidance), what parents see vs don't see, how parent
    visibility differs from coach visibility, support guidance
    philosophy ("support increases ownership, not replaces it"),
    example support scenarios, product value summary

Pipeline touchpoints:
- Stage 3 (expanded parent coaching message — new output
  alongside Coaching Message + Deep Dive)
- New output stream for Parent Insights Dashboard (reads from
  Interpretation JSON like Stage 4, but different compliance
  rules — parent-specific compliance framework needed)
- Stage 5 (Editorial Audit needs criteria for parent-facing
  content — different rules than athlete-facing or coach-facing)
- If Idea 1 adds ecosystem alignment data (parent involvement
  level, home conversation patterns), that data directly informs
  how the parent message is calibrated
- If Idea 2 adds daily coaching snippets, parents might want
  visibility into what their child is working on daily

Key design questions:
- Does the parent coaching message come from Stage 3 (same agent
  as athlete coaching) or a separate stage/sub-stage?
- What is the parent insights compliance framework? (Not NCAA,
  but still needs boundaries — what execution data is appropriate
  to show parents? All of it? A curated subset?)
- How does the parent message maintain the VirtusFocus voice
  while shifting audience from athlete to parent?
- Does the parent dashboard need its own audit criteria in
  Stage 5?
- How does parent visibility interact with athlete trust?
  (The document addresses this: journal content stays private)
- What is the right update cadence for parent insights?
  (Weekly like Coach Insights? More frequent?)

INTERACTION MAP — HOW THE 4 IDEAS CONNECT
============================================================

IDEA 1 → IDEA 2:
If intake captures ABI pillar scores (Ownership, Composure,
Focus, Structure), daily coaching snippets can target the
athlete's weakest pillar. New baseline fields give the daily
snippet more to work with.

IDEA 1 → IDEA 4:
If intake captures ecosystem alignment data (parent involvement
level, home conversation patterns), parent coaching messages
can be calibrated from day one. The Primary Problem Detector
output could inform parent support guidance.

IDEA 2 ↔ IDEA 3:
Both introduce daily processing. Must share daily data
consumption infrastructure. The daily coaching snippet
(athlete-facing) and daily coaching insights (coach-facing)
are different outputs from the same daily data stream.

IDEA 2 → IDEA 4:
If daily coaching snippets exist, parents might want visibility
into what their child is working on today. The parent dashboard
could surface the daily coaching focus area.

IDEA 3 → IDEA 4:
Daily coaching insights (coach-facing, compliance-restricted)
and parent insights (parent-facing, less restricted) are
parallel dashboard concepts with different compliance rules.
Design patterns may be shared.

ALL IDEAS:
All 4 ideas produce new output. All must maintain non-clinical
language compliance. All must be auditable (Stage 5 scope
expansion). All read from the Interpretation JSON or daily
app data. The versioning rules apply: new specification files
for new capabilities, source reference files edited in place.
