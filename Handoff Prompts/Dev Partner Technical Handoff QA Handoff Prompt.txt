=============================================================
WHAT THIS DOCUMENT IS
=============================================================
The Dev Partner Technical Handoff is a self-contained Markdown
document (~1,920 lines, ~13,600 words, 14 sections + glossary)
that tells an app development team EVERYTHING they need to
build the VirtusFocus backend. It covers:

- Intake form (29 questions with response formats)
- PPD deterministic scoring (4-layer algorithm, 8 buckets)
- ABI deterministic scoring (4 pillars, formula)
- Daily data collection and Mini-JSON assembly
- Weekly data assembly (Weekly Input Object)
- Execution signal computation (7 composite scores with
  complete formulas, 15 coach flags with trigger conditions)
- Stage 7 Daily Coach Signal (15-step deterministic rules)
- AI pipeline API interface (inputs/outputs/scheduling)
- Dashboard rendering (coach + parent)
- Data storage requirements
- Compliance rules
- Core Foundation compatibility
- Product tier activation matrix

The document was generated by reading 23 reference files and
synthesizing them into a developer-facing format. The risk is
that during synthesis, values may have been:
- Miscopied (wrong number, wrong enum value, wrong field name)
- Simplified incorrectly (formula component omitted or weight
  changed)
- Misattributed (wrong question mapped to wrong system)
- Omitted (edge case or threshold missing)
- Invented (value stated that doesn't exist in any source)
=============================================================
QA METHODOLOGY
=============================================================
This audit uses a LINE-BY-LINE verification methodology.
For every factual claim in the handoff document, trace it
back to the authoritative source file and confirm accuracy.

SEVERITY LEVELS:
- CRITICAL: Incorrect formula, wrong threshold, wrong enum
  value, missing required field, wrong field name, incorrect
  weight, wrong tier activation, incorrect timing rule, or any
  error that would cause the dev team to build something that
  produces wrong output. MUST be fixed before handoff.
- WARNING: Incomplete information that could cause confusion
  or require dev team to ask clarifying questions. Missing
  edge case, ambiguous phrasing, or inconsistent terminology.
  SHOULD be fixed before handoff.
- NOTE: Style, formatting, or organizational suggestions that
  improve clarity but don't affect correctness. NICE to fix.

QA PROCESS — SECTION BY SECTION:

For each of the 14 sections + glossary in the handoff
document:

1. READ the handoff document section in full
2. READ the corresponding authoritative source file(s) in full
3. COMPARE every factual claim:
   - Field names: exact match (case-sensitive, underscore-
     sensitive)
   - Enum values: exact match (every option listed, no extras,
     no missing, correct casing)
   - Formulas: every component present, correct weights,
     correct operations (round/floor/ceil), correct order
   - Thresholds/bands: exact boundary values, correct
     comparison operators (>= vs >)
   - Edge cases: all source edge cases present in handoff
   - Timing rules: correct sequencing, correct triggers
   - Question mappings: correct Q numbers to correct systems
   - Schema structures: correct nesting, correct types
4. DOCUMENT every discrepancy with:
   - Handoff document location (section + subsection)
   - What the handoff says (exact quote)
   - What the source says (exact quote with file + line)
   - Severity (Critical / Warning / Note)
   - Suggested correction
5. CONFIRM accuracy for sections that pass (brief statement)

CRITICAL RULES FOR THE QA SESSION
=============================================================
1. SOURCE FILES ARE TRUTH. If the handoff document says one
   thing and the source specification says another, the
   handoff document is WRONG. No exceptions.

2. READ THE FULL SOURCE. Do not skim source files. Formula
   verification requires reading every component definition.
   A single missed weight or threshold makes the formula
   wrong.

3. EXACT FIELD NAMES MATTER. "mindset_challenge_set_count"
   vs "mindset_challenge_accepted_count" is a CRITICAL error.
   The dev team will build database schemas from these names.

4. EXACT ENUM VALUES MATTER. "Partial Win" vs "Neutral" vs
   "Partial" is a CRITICAL error. The dev team will hard-code
   these strings.

5. EVERY NUMBER MATTERS. Weight of 30 vs 35, threshold of
   70 vs 75, range of 0-100 vs 1-100 — all CRITICAL errors.

6. EDGE CASES MATTER. If the source spec defines 12 edge
   cases for Stage 7 and the handoff only lists 10, that's
   a WARNING (2 missing edge cases the dev team won't handle).

7. DO NOT SUGGEST IMPROVEMENTS. This is a factual accuracy
   audit, not a content review. The handoff document's scope,
   organization, and writing style are out of scope. Only
   flag factual errors.

8. WORKED EXAMPLES: If the handoff includes worked examples,
   verify the arithmetic is correct by recomputing from the
   source formulas. Wrong arithmetic in examples is CRITICAL.

9. MISSING SECTIONS: If an entire topic from the source specs
   is absent from the handoff document, flag as WARNING with
   a note on what's missing.

10. ASSUME THE DEV TEAM IS LITERAL. They will implement
    exactly what the document says. If the document says
    "round to nearest integer" but the source says "floor",
    that's CRITICAL — it produces different outputs.
