VIRTUSFOCUS – INTERPRETATION ENGINE
VF_Interpretation_JSON_Rules.txt
(MASTER – LONGITUDINAL INTEGRATED + WEEKLY FORMAT RECOMMITMENT FIX)

PURPOSE
These rules standardize how weekly recap data is converted into structured JSON.
They reduce classification drift, enforce deterministic logic, and ensure downstream consistency
across weekly and longitudinal analysis.

All field classifications must follow these rules exactly.
If data does not support a classification, use "insufficient data".

============================================================
SECTION 1 – WIN THE DAY CLASSIFICATION RULES
============================================================

------------------------------------------------------------
1A) Daily Category Mapping (Numeric Scores)
------------------------------------------------------------

4–5 points → Win
2–3 points → Neutral
0–1 point  → Missed

Do not infer scores.
If numeric data missing, use category labels if provided.
If neither exists → "insufficient data".

APP TERMINOLOGY NOTE:
The app uses "Partial Win" as the athlete-facing label for the 2-3
range. "Neutral" remains the classification value in the
Interpretation JSON and all internal pipeline processing. Both map
to the same numeric range (2-3 points).

------------------------------------------------------------
1B) Weekly Tier (Numeric Total)
------------------------------------------------------------

28–35 total → Growth
14–27 total → Mixed
0–13 total  → Reset

------------------------------------------------------------
1C) Weekly Tier (Category-Only Fallback)
------------------------------------------------------------

If only category labels provided:

- 5+ Wins AND ≤1 Missed → Growth
- 2–4 Wins → Mixed
- 0–1 Wins → Reset

If total days incomplete → "insufficient data"

------------------------------------------------------------
1D) Consistency Profile
------------------------------------------------------------

Stable Positive:
- Majority Wins
- ≤1 Missed
- Low volatility across days

Mixed Pattern:
- Mixed distribution of Wins / Neutral / Missed
- No extreme swings

Volatile:
- Frequent swings Win ↔ Missed
- 2+ directional reversals across week

Reset Pattern:
- Majority Missed days
- No sustained rebound

------------------------------------------------------------
1E) Recommitment Signal (WITHIN-WEEK ONLY)
------------------------------------------------------------

Weekly recaps contain only 7 days.
There is no recorded day after Sunday.

Recommitment is evaluated ONLY when a Missed day has at least one subsequent day
inside the same weekly recap.

If the ONLY Missed day occurs on the final day of the week
(typically Sunday), and there is no earlier Missed day to evaluate recovery:
→ recommitment_signal = "insufficient data"

Deterministic Rules:

Strong:
- Any Missed day followed by a Win on the very next recorded day (within-week)
- OR a Missed day followed by a Win within the next 2 recorded days (within-week)
  AND the recap includes a clear corrective action statement

Moderate:
- Any Missed day followed by Neutral on the next recorded day (within-week)
- OR a Missed day followed by a Win later in the same week (delayed rebound) without clear corrective action

Low:
- Any Missed day followed by another Missed on the next recorded day (within-week)
- OR repeated Missed days with no rebound inside the week

Not Applicable:
- No Missed days in the recap

If multiple Missed events occur:
- If ANY event meets Strong → Strong
- Else if ANY event meets Moderate → Moderate
- Else if ANY event meets Low → Low
- Else → "insufficient data"

============================================================
SECTION 2 – BULLSEYE CLASSIFICATION RULES
============================================================

Primary Focus Ring:

Control:
- Emphasis on effort, habits, response, preparation

Influence:
- Mix of controllables + external factors

Cannot Control:
- Outcome fixation, referee, opponent, weather, coach decisions

Noise Fixation Present:

Yes:
- Repeated external attribution (3+ references)

No:
- Primary ownership language present

If unclear → "insufficient data"

============================================================
SECTION 3 – REFLECTION QUALITY SCALE
============================================================

1 → Surface description only (what happened)
2 → Basic awareness (no root cause linkage)
3 → Clear pattern recognition
4 → Ownership + forward behavioral adjustment

If no written reflection → "insufficient data"

============================================================
SECTION 4 – GROWTH PHASE CLASSIFICATION
============================================================

Emerging:
- Reactive patterns
- Inconsistent ownership
- Volatile execution

Developing:
- Increasing awareness
- Partial routine stability
- Inconsistent recommitment

Consistent:
- Stable execution
- Strong Win distribution
- Reliable corrective action

Leadership:
- Proactive regulation
- Influences teammates
- High ownership language
- Stable Growth tier pattern

If unclear → "insufficient data"

============================================================
SECTION 4C – GROWTH PHASE PROGRESSION THRESHOLDS
============================================================

Growth phase is NOT a static label assigned once at baseline.
It must be re-evaluated weekly using the criteria below.

Advancement requires sustained evidence across multiple weeks.
A single strong week does not trigger advancement.

If fewer than 2 prior weeks exist: growth_phase defaults to
the baseline classification from the athlete snapshot.

------------------------------------------------------------
4C-1) DEVELOPING → CONSISTENT
------------------------------------------------------------

Requires ALL of the following over 3+ weeks of data:

- rolling_win_rate ≥ 0.85 across the available window
- consistency_profile = Stable Positive for 2+ consecutive
  weeks (current week included)
- reflection_quality_score ≥ 3 for 2+ consecutive weeks
  (current week included)
- alignment_status = Aligned for 2+ consecutive weeks
  (current week included)
- If baseline_primary_derailer has been tested under adversity
  at any point in program history: execution held at the
  behavioral level (derailer may have activated, but daily
  Win status was maintained)

NOTE: If the derailer has NOT been tested yet (no adversity
trigger in program history), advancement to Consistent may
still occur if all other criteria are met for 4+ weeks.

------------------------------------------------------------
4C-2) CONSISTENT → LEADERSHIP
------------------------------------------------------------

Requires ALL of the following:

- All Consistent criteria sustained for 3+ consecutive weeks
  AFTER classification as Consistent
- reflection_quality_score = 4 at least once in the window
- Documented influence-ring behavior (teammate leadership,
  proactive support, mentoring) in at least 2 separate weeks
- Athlete language shows internal-standard motivation (naming
  personal standards rather than seeking external validation)

------------------------------------------------------------
4C-3) REGRESSION RULES
------------------------------------------------------------

Consistent → Developing:
- rolling_win_rate drops below 0.70 for 2+ consecutive weeks
- AND consistency_profile shifts to Mixed Pattern or Volatile

Leadership → Consistent:
- Influence-ring behavior absent for 3+ consecutive weeks
- AND reflection_quality_score drops below 3 for 2+
  consecutive weeks

Any phase → Emerging:
- Only if weekly_tier = Reset for 2+ consecutive weeks

------------------------------------------------------------
4C-4) MINIMUM DWELL TIME
------------------------------------------------------------

An athlete must spend at least 2 weeks in a phase before
advancing to the next phase. This prevents single-week
spikes from triggering premature phase changes.

The 2-week dwell clock starts from the week the phase
was first assigned (including baseline assignment).

------------------------------------------------------------
4C-5) INTEGRATION WITH TREND ANALYSIS
------------------------------------------------------------

When growth_phase changes from one week to the next:
- trend_analysis.growth_phase_movement = "Advancing" or
  "Regressing" as appropriate

Growth phase evaluation uses the same longitudinal window
as trend_analysis. If fewer than 2 prior weeks exist,
growth_phase_movement = "insufficient data".

============================================================
SECTION 4A – PRIOR COMMITMENT RESULT CLASSIFICATION RULES
============================================================

Completed:
- Recap contains specific behavioral evidence matching the prior action
- AND measurement target appears met or exceeded

Partial:
- Some behavioral evidence present
- BUT measurement target not fully met

Not Attempted:
- No behavioral evidence of the specific prior action in recap

Unknown:
- Recap data is ambiguous about this specific behavior
- OR prior action cannot be clearly assessed from recap format

No Prior Week:
- First week of program; no prior JSON exists

Do not infer execution from general behavior.
The evidence must specifically match the prior micro_commitment.action.

============================================================
SECTION 4B – EMOTIONAL INTENSITY CLASSIFICATION RULES
============================================================

Low:
- Task-oriented, neutral language throughout
- No emotional references beyond routine satisfaction/frustration

Moderate:
- Athlete explicitly names friction, frustration, or strain
- BUT execution was not disrupted
- AND language remains largely composed

High:
- Significant emotional strain explicitly described
- Examples: mental breakdown language, homesickness, strong anger,
  identity-level crisis statements, desire to quit/leave
- Execution may or may not have been disrupted

If unclear from recap language: "insufficient data".

This is a language signal, not a clinical assessment.

============================================================
SECTION 5 – LONGITUDINAL TREND CLASSIFICATION RULES
============================================================

Applies only if at least 2 weeks of prior data supplied.
If fewer than 2 comparison points → "insufficient data"

------------------------------------------------------------
5A) Weekly Tier Trend
------------------------------------------------------------

Improving:
- Movement Reset → Mixed
- Mixed → Growth
- Reset → Growth (2-level jump)

Stable:
- Same tier consecutive weeks

Declining:
- Growth → Mixed
- Mixed → Reset
- Growth → Reset (2-level drop)

------------------------------------------------------------
5B) Growth Phase Movement
------------------------------------------------------------

Advancing:
- Emerging → Developing
- Developing → Consistent
- Consistent → Leadership

Stable:
- Same phase consecutive weeks

Regressing:
- Any downward phase movement

If phase data missing → "insufficient data"

------------------------------------------------------------
5C) Consistency Trend
------------------------------------------------------------

Strengthening:
- Movement toward Stable Positive
- Reduction in volatility across weeks

Volatile:
- Shifting between Volatile / Mixed / Reset Pattern across weeks

Stagnant:
- Same non-Stable pattern across 3 weeks
- No measurable stabilization

------------------------------------------------------------
5D) Recommitment Trend
------------------------------------------------------------

Escalating:
- Low → Moderate → Strong progression
- Visible faster recovery timing week over week (when within-week recovery data exists)

Stable:
- Same recommitment classification consecutive weeks

Weakening:
- Strong → Moderate → Low progression
- Slower recovery timing (when within-week recovery data exists)

If one or more weeks are "insufficient data" due to end-of-week Missed:
- recommitment_trend may become "insufficient data" unless at least 2 comparable weeks exist

------------------------------------------------------------
5E) Pattern Shift Detection
------------------------------------------------------------

Yes:
- core_pattern changes category
- behavioral_root_cause changes classification
- most_important_controllable materially shifts focus area

No:
- Pattern remains structurally consistent

If insufficient historical anchor → "insufficient data"

SECTION 7 – MORNING TUNE-UP CLASSIFICATION RULES
============================================================

All rules in this section apply ONLY when input_source = "app".

------------------------------------------------------------
7A) Counting Rules
------------------------------------------------------------

days_completed:
  count(morning_tune_up.completed == true) across 7 daily events

days_missed:
  7 - days_completed

completion_rate:
  days_completed / 7

------------------------------------------------------------
7B) Morning Timing Classification
------------------------------------------------------------

Stage 2 classifies each completed Morning Tune-Up by comparing
completion_timestamp against the morning cutoff.

Morning cutoff: 12:00 PM (noon) local time.

on_time:
  completion_timestamp before 12:00 PM local time

late:
  completion_timestamp at or after 12:00 PM local time
  AND completed = true

on_time_count:
  count(on-time completions) across 7 days

late_count:
  count(late completions) across 7 days

Not completed (completed = false): excluded from both counts.
on_time_count + late_count = days_completed.

------------------------------------------------------------
7C) Timeliness Profile
------------------------------------------------------------

on_time_rate = on_time_count / days_completed

Proactive:
- on_time_rate >= 0.71
- Majority of completed Tune-Ups before noon

Delayed:
- on_time_rate >= 0.30 AND < 0.71
- Mix of on-time and late completions

Reactive:
- on_time_rate < 0.30
- Majority of completed Tune-Ups after noon

If days_completed == 0 → "insufficient data"

------------------------------------------------------------
7D) Mindset Challenge Metrics
------------------------------------------------------------

mindset_challenge_accepted_count:
  count(morning_tune_up.mindset_challenge_accepted == true)
  across 7 daily events

V1 MANDATORY ACCEPTANCE RULE:
In V1, the Mindset Challenge checkbox is required to complete
the Tune-Up. Therefore:
- mindset_challenge_accepted = true when completed = true
- mindset_challenge_accepted = null when completed = false
  (athlete did not encounter the challenge)
- mindset_challenge_accepted_count = days_completed in V1

mindset_challenge_completed_count:
  count(evening_review.mindset_challenge_completed == true)
  across 7 daily events

mindset_challenge_follow_through_rate:
  mindset_challenge_completed_count /
    max(mindset_challenge_accepted_count, 1)

If mindset_challenge_accepted_count == 0 AND
days_completed == 0 → "insufficient data"

V1 DOWNSTREAM IMPACT ON FOLLOW-THROUGH SCORE:
Because V1 acceptance is mandatory, Follow-Through Score C2
(Challenge Acceptance Rate = accepted_count / days_completed)
is always 1.00 for completed Tune-Ups. The C2 weight (30%)
effectively rewards Tune-Up completion itself. This is
acceptable in V1. If future versions make acceptance optional,
C2 will differentiate between athletes who accept challenges
and those who decline.

------------------------------------------------------------
7E) Longitudinal Streak Rules
------------------------------------------------------------

current_streak:
  Count consecutive completed Morning Tune-Up days backward
  from the most recent day (Sunday of current week), crossing
  week boundaries via prior week JSONs.
  - completed = true continues the streak
  - completed = false breaks the streak
  - Count stops at the first uncompleted day
  If no prior week JSON exists: computed from current week
  only (maximum 7).

longest_streak:
  Maximum current_streak value across all prior week JSONs
  plus current week.
  If no prior week JSON exists: longest_streak = current_streak.

Both fields follow the same longitudinal computation pattern
as rolling_win_rate_4wk (reads prior week JSONs).

SECTION 8 – EVENING REVIEW CLASSIFICATION RULES
============================================================

All rules in this section apply ONLY when input_source = "app".

------------------------------------------------------------
8A) Review Completion Counting
------------------------------------------------------------

Full Completion:
- All 3 components completed (wtd.completed AND
  journaling.completed AND bullseye.completed)
- AND evening_review.submitted = true

Partial Completion:
- At least 1 component completed but NOT full completion
- Includes submitted reviews with incomplete components
- Includes non-submitted reviews (submitted = false) where
  at least 1 component has data (hard-out lockout scenario)

Missed:
- No components completed (all 3 component completed = false)
- AND no partial data present

full_completion_count:
  count(full completion days) across 7 days

partial_completion_count:
  count(partial completion days) across 7 days

missed_count:
  count(missed days) across 7 days

full_completion_count + partial_completion_count +
missed_count = 7

completion_rate:
  full_completion_count / 7
  (FULL completions only in numerator)

------------------------------------------------------------
8B) Evening Timing Classification
------------------------------------------------------------

Four-category timing model. Stage 2 classifies each submitted
Evening Review by comparing completion_timestamp against
cutoffs.

HARD-OUT TIME PARAMETER:
Stage 2 receives the hard-out time as a program-level
configuration parameter.
Default: 6:00 AM local time (next day).
Configurable per program.
The hard-out time is when the app locks the previous day's
Evening Review and releases the next day's Morning Tune-Up.

BACKFILL BOUNDARY:
hard-out time minus 2 hours.
Fixed constant. Not configurable.

CATEGORIES:

On-time:
  completion_timestamp before 10:00 PM local time
  Same-evening routine engagement

Late:
  completion_timestamp at or after 10:00 PM AND before
  backfill boundary
  Same-night, after intended window

Backfill:
  completion_timestamp at or after backfill boundary AND
  before hard-out time
  Next-morning retroactive catch-up before lockout

Missed (timing):
  Evening Review not submitted by hard-out time
  submitted = false. No completion_timestamp.

DEFAULT CUTOFFS (6:00 AM hard-out):
  On-time:  before 10:00 PM
  Late:     10:00 PM to 4:00 AM
  Backfill: 4:00 AM to 6:00 AM
  Missed:   after 6:00 AM (locked out)

on_time_count:
  count(on-time submissions) across 7 days

late_submission_count:
  count(late submissions) across 7 days

backfill_count:
  count(backfill submissions) across 7 days

Only submitted reviews (submitted = true) are classified into
timing categories. Non-submitted days are excluded from all
three timing counts.

on_time_count + late_submission_count + backfill_count =
  count(submitted reviews)

------------------------------------------------------------
8C) Trigger Method Classification
------------------------------------------------------------

self_initiated_count:
  count(evening_review.trigger_method == "self_initiated")
  across 7 days

reminder_dependent_count:
  count(evening_review.trigger_method == "reminder")
  across 7 days

trigger_method is tagged by the app at point of entry.
Stage 2 trusts this tag without re-derivation.

Days with trigger_method = null (no engagement started):
excluded from both counts.

------------------------------------------------------------
8D) Sequence Integrity Classification
------------------------------------------------------------

Designed sequence: WTD → Journaling → Bullseye

For each day with 2+ components completed (component_sequence
contains 2+ entries), compare actual component_sequence against
designed order.

multi_component_days:
  count(days with 2+ entries in component_sequence array)

designed_order_days:
  count(multi_component_days where component_sequence matches
  designed order exactly or is a subsequence of designed order)

adherence_rate = designed_order_days / multi_component_days

Intact:
- adherence_rate >= 0.86
- At most 1 deviation from designed sequence

Partial:
- adherence_rate >= 0.43 AND < 0.86
- Some deviations but designed order followed in plurality

Broken:
- adherence_rate < 0.43
- Majority of multi-component reviews deviated

If multi_component_days < 2 → "insufficient data"

Days with only 1 component completed are excluded from
sequence evaluation (sequence requires 2+ components).

------------------------------------------------------------
8E) Component Completion Counting
------------------------------------------------------------

component_completion.wtd_completed_count:
  count(wtd.completed == true
    AND evening_review.submitted == true)

component_completion.journaling_completed_count:
  count(journaling.completed == true
    AND evening_review.submitted == true)

component_completion.bullseye_completed_count:
  count(bullseye.completed == true
    AND evening_review.submitted == true)

SUBMITTED-ONLY RULE: Component completion counts include ONLY
days where evening_review.submitted = true. Non-submitted
reviews (hard-out lockout, submitted = false) are excluded
from component counts even if partial component data exists.

------------------------------------------------------------
8F) Partial Data from Hard-Out Lockout
------------------------------------------------------------

When the hard-out locks an incomplete Evening Review:
- submitted = false
- completion_timestamp = null
- Component data entered before lockout is preserved in the
  daily event record
- The day counts as PARTIAL in partial_completion_count
  (athlete engaged with at least 1 component)
- The day has no timing classification (excluded from
  on_time, late, backfill counts)
- Component data does NOT count toward component_completion
  fields (wtd_completed_count, journaling_completed_count,
  bullseye_completed_count) — SUBMITTED-ONLY RULE applies
- Component TEXT data IS available for content analysis by
  existing v9.3 fields (athlete_state, core_pattern,
  emotional_intensity, reflection_quality_score,
  bullseye_analysis)

Partial data informs WHAT the athlete thought/felt (v9.3
content signals). It does NOT credit HOW the athlete used the
system (execution behavior metrics).

SECTION 9 – WTD QUESTION PATTERN CLASSIFICATION RULES
============================================================

All rules in this section apply ONLY when input_source = "app".

------------------------------------------------------------
9A) Per-Question Yes Rates
------------------------------------------------------------

wtd_completed_days = count(days where wtd.completed == true)

For each question (q1 through q5):
  yes_rate = count(q_n == true) / wtd_completed_days

Rates are decimal 0.00-1.00. Only days with WTD completed
are included in the denominator.

If wtd_completed_days == 0 → all yes rates = "insufficient data"

------------------------------------------------------------
9B) Weakest and Strongest Question
------------------------------------------------------------

weakest_question:
  The question (Q1-Q5) with the lowest yes_rate.

strongest_question:
  The question (Q1-Q5) with the highest yes_rate.

Tie rules:
- If 2+ questions share the lowest yes_rate →
  weakest_question = "none"
- If 2+ questions share the highest yes_rate →
  strongest_question = "none"
- If all 5 rates are equal → both = "none"

If wtd_completed_days < 3 → both = "insufficient data"

------------------------------------------------------------
9C) Intraweek Volatility
------------------------------------------------------------

Computed from the sequence of WTD daily categories across
consecutive completed days within the week.

Daily categories (from Section 1A):
  4-5 = Win, 2-3 = Neutral, 0-1 = Missed

category_changes:
  Count of transitions between DIFFERENT WTD daily categories
  across consecutive completed days (chronological order).
  Each transition where the category differs from the previous
  completed day = 1 change. Same category = no change.

Low:
- category_changes <= 1
- Scores stay within the same or adjacent categories

Moderate:
- category_changes = 2 or 3
- Some bouncing between categories

High:
- category_changes >= 4
- Frequent swings between categories across the week

If wtd_completed_days < 3 → "insufficient data"

------------------------------------------------------------
9D) Recovery Speed Days
------------------------------------------------------------

Measures days from a Missed day (WTD daily score 0-1) to the
next Win day (WTD daily score 4-5) within the same week.

Counting:
- Start day = Missed day (score 0-1)
- End day = next Win day (score 4-5), chronological order
- recovery_speed_days = calendar days between start and end
  (inclusive of end, exclusive of start)
  Monday Missed → Tuesday Win = 1 day
  Monday Missed → Wednesday Win = 2 days
  Monday Missed → Sunday Win = 6 days

Multiple Missed events:
  Use the FASTEST recovery instance (lowest count).
  Consistent with recommitment_signal logic (Section 1E).

Values:
  integer 1-7: recovery occurred within-week
  "not applicable": no Missed days in the week
  "did not recover": Missed day occurred but no subsequent
    Win day within the week (excluding last-day-only Missed)
  "Pending Data": the ONLY or LAST Missed day occurs on the
    final day of the week with no subsequent day. Recovery
    deferred to next week via cross_week_recovery.
  "insufficient data": WTD data unavailable

"Pending Data" is distinct from "insufficient data":
  "Pending Data" = measurement deferred, data WILL arrive
  "insufficient data" = data genuinely absent

------------------------------------------------------------
9E) Cross-Week Recovery
------------------------------------------------------------

Resolves the blind spot created by end-of-week Missed days.

prior_week_unresolved_miss:
  "Yes" → prior week JSON recovery_speed_days = "Pending Data"
  "No" → prior week recovery_speed_days != "Pending Data"
  Requires reading prior week JSON.

recovery_days (cross-week):
  When prior_week_unresolved_miss = "Yes":
    Count days from Monday of current week until first Win
    day (score 4-5):
      Monday Win    = 1
      Tuesday Win   = 2
      Wednesday Win = 3
      Thursday Win  = 4
      Friday Win    = 5
      Saturday Win  = 6
      Sunday Win    = 7
      No Win all week = "did not recover"

  When prior_week_unresolved_miss = "No":
    recovery_days = "not applicable"

If no prior week JSON (first week of program):
  prior_week_unresolved_miss = "No"
  recovery_days = "not applicable"

Core Foundation fallback:
  prior_week_unresolved_miss =
    "not available — weekly recap input only"
  recovery_days =
    "not available — weekly recap input only"

The prior week JSON is NEVER modified. Forward-only data flow.

SECTION 10 – JOURNALING BEHAVIOR CLASSIFICATION RULES
============================================================

All rules in this section apply ONLY when input_source = "app".

These rules classify BEHAVIORAL execution quality of journaling
— length and domain coverage. This is distinct from
reflection_quality_score (Section 3), which measures CONTENT
quality (depth of insight and pattern recognition).

An athlete can have high RQS + Compressed depth_profile
(insightful but rushed) or low RQS + Thorough depth_profile
(lengthy but surface-level).

------------------------------------------------------------
10A) Depth Profile
------------------------------------------------------------

avg_daily_word_count:
  total words across ALL journal entries for the week
  / journaling_completed_days

Stage 2 computes word counts from journaling entry text fields
(school_work_entry, sport_entry, home_life_entry).

journaling_completed_days:
  count(days where journaling.completed == true AND
  evening_review.submitted == true)

Thorough:
- avg_daily_word_count >= 75
- Substantial engagement across completed days

Adequate:
- avg_daily_word_count >= 30 AND < 75
- Meaningful entries but not extensive

Compressed:
- avg_daily_word_count >= 10 AND < 30
- Brief entries suggesting rushed engagement

Minimal:
- avg_daily_word_count < 10
- Negligible text volume

If journaling_completed_days == 0 → "insufficient data"

------------------------------------------------------------
10B) Compression Detected
------------------------------------------------------------

Compression is a longitudinal signal: the athlete's journaling
depth has declined relative to their own prior baseline.

prior_2wk_avg = average of avg_daily_word_count from the
  2 most recent prior week JSONs with journaling data

Yes:
- avg_daily_word_count < (prior_2wk_avg x 0.70)
- Current week at least 30% below 2-week prior average

No:
- avg_daily_word_count >= (prior_2wk_avg x 0.70)
- No meaningful decline from prior baseline

If fewer than 2 prior weeks with journaling data →
  "insufficient data"

------------------------------------------------------------
10C) Domain Omission Pattern
------------------------------------------------------------

For each domain (school_work, sport, home_life), compute
total weekly word count across all journaling-completed days.

A domain is "omitted" if its total weekly word count < 10
words across all completed journaling days.

none:
- No domain omitted (all 3 domains have >= 10 words total)

school_work:
- school_work domain omitted, other domains active

sport:
- sport domain omitted, other domains active

home_life:
- home_life domain omitted, other domains active

multiple:
- 2+ domains omitted

If journaling_completed_days < 3 → "insufficient data"

SECTION 11 – BULLSEYE BEHAVIOR CLASSIFICATION RULES
============================================================

All rules in this section apply ONLY when input_source = "app".

------------------------------------------------------------
11A) Completion Reliability
------------------------------------------------------------

bullseye_completed_count:
  count(bullseye.completed == true AND
  evening_review.submitted == true) across 7 days

Consistent:
- bullseye_completed_count >= 6

Partial:
- bullseye_completed_count >= 3 AND <= 5

Inconsistent:
- bullseye_completed_count <= 2

If no evening engagement days (full_completion_count +
partial_completion_count == 0) → "insufficient data"

------------------------------------------------------------
11B) Ring Dominance Rates
------------------------------------------------------------

For each day where Bullseye was completed AND submitted:

center_ring_dominant day:
  center_ring_items.length > influence_ring_items.length
  AND center_ring_items.length > outer_ring_items.length

outer_ring_dominant day:
  outer_ring_items.length > center_ring_items.length
  AND outer_ring_items.length > influence_ring_items.length

bullseye_completed_days:
  count(days with bullseye.completed == true AND
  evening_review.submitted == true)

center_ring_dominant_rate:
  count(center_ring_dominant days) / bullseye_completed_days

outer_ring_dominant_rate:
  count(outer_ring_dominant days) / bullseye_completed_days

Both rates are decimal 0.00-1.00.

If bullseye_completed_days == 0 → both = "insufficient data"

Ties (no single ring has strictly more items than both
others): that day is neither center-dominant nor
outer-dominant.

------------------------------------------------------------
11C) Ring Balance
------------------------------------------------------------

Center-Weighted:
- center_ring_dominant_rate >= 0.60
- Center ring dominates on 60%+ of Bullseye days

Outer-Weighted:
- outer_ring_dominant_rate >= 0.40
- Outer ring dominates on 40%+ of Bullseye days
- ASYMMETRIC THRESHOLD: Outer-ring dominance at 0.40 is
  already a coaching signal (athlete focusing on
  uncontrollables nearly half the time)

Balanced:
- Neither Center-Weighted nor Outer-Weighted threshold met

Both thresholds cannot be met simultaneously (a day has at
most one dominant ring, so rates cannot sum above 1.00).

If bullseye_completed_days == 0 → "insufficient data"

------------------------------------------------------------
11D) Contradiction Detected
------------------------------------------------------------

Cross-references WTD daily category with Bullseye ring
dominance on the same day.

A contradictory day occurs when:

Category A: WTD daily category = Win (4-5 score)
  AND outer_ring_items is the largest ring
  (more items than BOTH center and influence)
  — athlete claims a winning day but focused on
  uncontrollables

Category B: WTD daily category = Missed (0-1 score)
  AND center_ring_items is the largest ring
  (more items than BOTH outer and influence)
  — athlete claims a missed day but says everything
  was in their control

contradiction_days = count(days with Category A or B)

Yes:
- contradiction_days >= 2

No:
- contradiction_days < 2

If fewer than 3 days with both WTD completed AND Bullseye
completed (and submitted) → "insufficient data"

Moderate: rating >= 4 AND <= 7

------------------------------------------------------------
12B) Confidence Alignment
------------------------------------------------------------

Confidence compares against Ownership Index and Follow-Through
Score — the two scores most aligned with self-perceived

  Count top-band scores:

    follow_through_score.band == "Strong" → +1 top

habit_execution_tier:

habit_perception_tier:
  From habit_consistency_level using 12A rules

Aligned:
- habit_perception_tier == habit_execution_tier

Conflated:
- habit_perception_tier > habit_execution_tier

Undervalued:
- habit_perception_tier < habit_execution_tier

If habit_consistency_level = null → "insufficient data"
If rhythm_score.band = "insufficient data" →
  "insufficient data"

SECTION 18 – FORWARD ANCHOR CLASSIFICATION RULES
============================================================

The forward_anchor block captures the athlete's weekly
controllable intention ("One thing you can control next week")
and produces three analytical signals: controllability
classification, focus consistency, and forward anchor alignment.

All rules apply ONLY when input_source = "app" AND
forward_anchor.submitted = true.

Core Foundation fallback:
  text: null
  submitted: false
  controllability_classification: "not available"
  prior_forward_anchor: {
    prior_text: null,
    alignment_status: "not available",
    alignment_notes: "not available — core foundation athlete"
  }
  focus_consistency: "not available"

When input_source = "app" AND forward_anchor.submitted = false:
  controllability_classification: "insufficient data"
  prior_forward_anchor: computed normally (prior week may exist)
  focus_consistency: current week does not contribute; evaluate
    from prior weeks only if applicable, else "insufficient data"

------------------------------------------------------------
18A) Controllability Classification
------------------------------------------------------------

Agent-classified from forward_anchor.text content. Same
classification pattern as PPD tie-breaker (agent judgment
with guided examples, not pure keyword matching).

Three categories:

  Controllable
    The stated intention is an action, behavior, or mindset
    approach within the athlete's direct control.
    Indicators: action verbs (respond, complete, bring, focus,
    prepare, commit, practice), behavioral specificity, effort-
    oriented language, process focus.
    Examples:
      "Respond calmly to mistakes" → Controllable
      "Complete my morning challenge every day" → Controllable
      "Bring energy to practice" → Controllable
      "Stay locked in during drills" → Controllable
      "Do my full warmup before every game" → Controllable

  Outcome-Oriented
    The stated intention is a result the athlete cannot directly
    control. Depends on external factors, opponents, selection
    decisions, or scoring.
    Indicators: outcome language (win, score, get selected, beat,
    earn, achieve a result), external dependency, comparative
    framing against others.
    Examples:
      "Win my next match" → Outcome-Oriented
      "Score in every game" → Outcome-Oriented
      "Get more playing time" → Outcome-Oriented
      "Be the best on the team" → Outcome-Oriented

  Mixed
    Contains both controllable and outcome elements, or the
    controllability is ambiguous. The athlete is reaching toward
    something within their control but frames it through an
    outcome lens (or vice versa).
    Examples:
      "Play confident in the big moments" → Mixed
        (confidence is internal state, not directly controllable
        like a behavior, but "play" implies effort)
      "Be a good teammate" → Mixed
        (controllable intent, but vague — no specific behavior)
      "Trust my preparation and let the results come" → Mixed
        (trust = controllable mindset, results = outcome)
      "Give 100% so I can earn my spot" → Mixed
        (effort = controllable, earning spot = outcome)

  Classification requires submitted = true AND non-empty text.
  If submitted = false or text is null → "insufficient data".

  LONGITUDINAL SIGNIFICANCE: A pattern of Outcome-Oriented →
  Controllable over weeks indicates growth aligned with the
  Bullseye Method's center-ring philosophy. Stage 2 does not
  compute this longitudinal pattern explicitly — it is visible
  in the forward_anchor history across weekly JSONs and may
  inform Stage 3 and Stage 4 downstream.

------------------------------------------------------------
18B) Focus Consistency Classification
------------------------------------------------------------

Agent-classified cross-week comparison of forward_anchor text
themes. Requires the current week AND at least one prior week
with forward_anchor.submitted = true.

Three categories:

  Consistent
    Current anchor shares the same theme, behavioral domain,
    or closely related controllable focus as prior week(s).
    The athlete is sustaining attention on the same area.
    Example: Week N = "Bring energy to practice."
             Week N+1 = "Show up with effort every day."
    Same theme: effort and energy in practice. → Consistent

  Shifting
    Current anchor addresses a different behavioral domain or
    focus area than the prior week. Not inherently negative —
    may reflect appropriate adaptation to new context.
    Example: Week N = "Respond calmly to mistakes."
             Week N+1 = "Complete my morning challenge every day."
    Different domain: mistake response vs. routine compliance.
    → Shifting

  Scattered
    No coherent thread across 3+ consecutive submitted anchors.
    Each week's focus is unrelated to the others. Requires 3+
    weeks of submitted anchor data to classify.
    Example: Week N = "Win my next match."
             Week N+1 = "Be nicer to teammates."
             Week N+2 = "Sleep more."
    No thread. → Scattered

  If fewer than 2 submitted anchors available (current + at
  least 1 prior) → "insufficient data".

  If current anchor not submitted but 2+ prior anchors exist →
  "insufficient data" (cannot classify without current data).

  CONTEXTUAL NOTE: Consistent focus is not always better than
  Shifting. An athlete who shifts focus after resolving a prior
  area is showing adaptive attention. Scattered focus with no
  follow-through is the coaching signal. Stage 2 classifies
  only — interpretation of whether the pattern is positive or
  negative is a Stage 3/4 downstream concern.

------------------------------------------------------------
18C) Forward Anchor Alignment (Deferred Signal)
------------------------------------------------------------

Deferred one-week signal. Same timing pattern as
prior_commitment_result.

The alignment is computed in the CURRENT week's JSON by
looking back at the PRIOR week's forward_anchor.text and
comparing it against the CURRENT week's daily behavior.

Input: Prior week JSON's forward_anchor.text
Evidence sources (current week daily behavior):
  - Morning Tune-Up focus words (daily_events[].morning_tune_up.
    focus_word) — thematic alignment with stated anchor
  - Bullseye entries (daily_events[].evening_review.bullseye
    ring items) — center-ring items reflecting anchor theme
  - Journal reflections (daily_events[].evening_review.journaling
    entries) — references to stated intention or its behavioral
    domain
  - WTD question patterns — Q1 (intention) and Q3 (adversity
    response) patterns that reflect anchor execution

Classification:

  Aligned
    Behavioral evidence across 3+ days shows the athlete
    acting on their stated intention. Focus words, Bullseye
    center-ring items, or journal entries reference the anchor's
    theme or behavioral domain.

  Partial
    Some behavioral evidence (1-2 days or weak thematic
    connection) but inconsistent. The athlete showed awareness
    of the intention but did not sustain it through the week.

  Misaligned
    No behavioral evidence of the stated intention in daily
    data. The anchor had no apparent connection to the athlete's
    actual behavior during the following week.

  No Prior Week
    First week in the program, OR the prior week's
    forward_anchor.submitted = false. No anchor to evaluate.

  insufficient data
    Prior anchor exists but current week daily data is
    insufficient to evaluate (e.g., very few completed days,
    minimal journal/Bullseye content).

  Alignment notes: 1-2 sentence evidence-based explanation.
  Same style as prior_commitment_result.execution_notes.
  Example: "Prior anchor 'Respond calmly to mistakes' showed
  alignment in 4/6 journal entries referencing composure after
  errors and 3 center-ring Bullseye items about response quality."

------------------------------------------------------------
18D) Cross-Week Data Access
------------------------------------------------------------

Forward Anchor classification requires prior week JSON access
for two of three signals:

  focus_consistency:
    Read prior week(s) forward_anchor.text and submitted status.
    For Scattered classification: read 2+ prior weeks.
    No prior week → "insufficient data".

  prior_forward_anchor.alignment_status:
    Read prior week JSON's forward_anchor.text.
    No prior week → "No Prior Week".
    Prior week exists but submitted = false → "No Prior Week".

  controllability_classification:
    Current week only. No cross-week data needed.

This follows the same cross-week access pattern as
trend_analysis, coach_flags 13-15, and cross_week_recovery.

------------------------------------------------------------
18E) Validation Rules
------------------------------------------------------------

  forward_anchor.text:
    String or null. No length validation (athlete-authored).

  forward_anchor.submitted:
    Boolean. true when text is non-null and non-empty.

  controllability_classification:
    Must be one of: Controllable | Outcome-Oriented | Mixed |
    insufficient data | not available

  prior_forward_anchor.prior_text:
    String or null.

  prior_forward_anchor.alignment_status:
    Must be one of: Aligned | Partial | Misaligned |
    No Prior Week | insufficient data | not available

  prior_forward_anchor.alignment_notes:
    String. "No prior anchor to evaluate" when alignment_status
    = "No Prior Week". Evidence-based for all other statuses.

  focus_consistency:
    Must be one of: Consistent | Shifting | Scattered |
    insufficient data | not available

SECTION 19 – FORWARD ANCHOR AND MICRO-COMMITMENT RELATIONSHIP
============================================================

The forward_anchor (athlete-authored intention) and
micro_commitment (coach-assigned action, Step 10) are distinct
but related fields.

DESIGN DECISION (DQ1 — Anchor-Informed Commitment):
The micro_commitment BUILDS ON the athlete's Forward Anchor
when alignment is strong. This is the approved interaction
model for all pipeline stages.

STAGE 2 RESPONSIBILITIES:
  - Extract and classify forward_anchor (Section 18)
  - Generate micro_commitment (Step 10) independently using
    existing rules
  - Store both fields in the output JSON
  - Stage 2 does NOT implement the anchor-informed interaction —
    that is a STAGE 3 responsibility (micro_commitment framing
    informed by forward_anchor context)

The forward_anchor.text is stored in the Interpretation JSON
for downstream consumption. Stage 3 reads both the
forward_anchor and micro_commitment fields and applies the
anchor-informed interaction model when generating the Coaching
Message.

RAW TEXT PRIVACY RULE:
forward_anchor.text is athlete-private. It informs Stages 3,
4, and 6 but is NEVER surfaced to coaches or parents. Same
compliance protection as journal content. Derived
classifications (controllability, consistency, alignment) are
compliance-safe in translated behavioral language.
