SECTION 1: CURRENT STATE SUMMARY

1.1 WHAT STAGE 2 READS FROM THE SNAPSHOT TODAY

The Interpretation Engine (v9.4) reads exactly 5 fields from the baseline snapshot at Step 0-1 of the Core Interpretation Process:

    baseline_profile:
      baseline_identity_claim    (prose sentence)
      baseline_hinge_habit       (prose sentence)
      baseline_primary_derailer  (prose sentence)
      baseline_why_it_matters    (prose sentence)
      baseline_growth_phase      (enum: Emerging|Developing|Consistent|Leadership)

These 5 fields are:
- Read from the snapshot file on disk
- Written unchanged to the weekly Interpretation JSON (baseline_profile block)
- Consumed by all downstream stages (3, 4, 5) FROM THE JSON, not from the snapshot
- Used internally by Stage 2 for:
  - baseline_alignment assessment (Step 8.5 — compares weekly patterns against baseline anchors)
  - Growth phase progression evaluation (Section 4C — checks derailer testing for Developing->Consistent)
  - Narrative arc generation (uses identity claim and derailer as longitudinal reference points)

1.2 HOW DOWNSTREAM STAGES CONSUME BASELINE DATA

All downstream stages read the Interpretation JSON. None read the snapshot directly.

Stage 3 (Coaching Output Engine v1.5):
- baseline_identity_claim -> identity threading in coaching messages
- baseline_hinge_habit -> micro-commitment grounding
- baseline_primary_derailer -> derailer address in coaching
- baseline_growth_phase -> growth phase-appropriate tone

Stage 4 (Coach Insights Engine v1.2):
- baseline_profile -> contextual reference for developmental framing
- baseline_growth_phase -> developmental context for institutional output
- baseline_alignment -> informs Section 2 (Weekly Performance Context)

Stage 5 (Editorial Audit v1.1):
- baseline_profile -> verifies coaching output consistency with baseline anchors
- baseline_alignment -> checks alignment_status matches coaching narrative

1.3 THE INTEGRATION GAP

The v2.0 App athlete snapshot introduces 40+ new structured fields across 6 data sub-blocks that have NO LANDING ZONE in the current weekly JSON schema:

GROUP A — New Header Fields: COMPETITIVE_LEVEL, SEASON_PHASE, COMMITMENT_LEVEL
GROUP B — PPD_SCORES: ppd_primary_problem, ppd_top_3, ppd_all_scores, ppd_tie_breaker_classification
GROUP C — ABI_SCORES: pillar scores, total, band, primary_emphasis, pillar_bands
GROUP D — ECOSYSTEM: parent_involvement_level, home_conversation_pattern, support_network_primary, parent_inclusion, parent_email
GROUP E — BEHAVIORAL_PATTERNS: adversity_response_pattern, adversity_self_description
GROUP F — LONGITUDINAL_BASELINES: season_phase, reset_speed_baseline, competitor_aspiration_text, mental_game_self_rating

None of these fields appear in the v9.4 canonical output schema. None have classification rules. None have processing steps. The entire PPD/ABI/ecosystem/behavioral/longitudinal layer from the expanded snapshot currently has no path into the weekly JSON and therefore no path to downstream stages.

SECTION 2: FIELD-BY-FIELD ASSESSMENT
============================================================

Assessment categories:
- ACTIVE USE: Stage 2 reads the field and uses it in a classification rule or computation. Gets its own JSON field with a derivation rule.
- PASS-THROUGH: Stage 2 reads the field from the snapshot and copies it into the weekly JSON unchanged. No classification rule needed. Downstream stages consume it from the JSON.
- NO ACTION: Field stays in the snapshot only. Stage 2 does not read it. Not carried in the weekly JSON.

------------------------------------------------------------
GROUP A — New Header Fields
------------------------------------------------------------

COMPETITIVE_LEVEL
  Source: Q4 (Youth / Middle School / High School / College / Professional)
  Recommendation: PASS-THROUGH
  Rationale: Stage 3 coaching calibration input. Competitive level determines coaching
  tone, vocabulary complexity, directness level, and accountability framing intensity.
  A middle school athlete and a college athlete with identical data patterns require
  fundamentally different coaching voices. This is a scalability-critical field — the app
  must serve athletes across all competitive levels with appropriately calibrated output.
  Stage 3 must implement a formal Competitive Level Calibration Rule (flagged for Task 7).
  Stage 4 may use it for contextual framing in institutional output.
  CF fallback: "not available — core foundation intake"

SEASON_PHASE (intake value)
  Source: Q6 (Off-Season / Pre-Season / In-Season / Post-Season / Year-Round Training)
  Recommendation: PASS-THROUGH as "baseline_season_phase" (RENAMED)
  Rationale: This is POINT-IN-TIME INTAKE DATA, not a live indicator. An athlete who
  fills out their intake during Pre-Season may be 6 weeks into In-Season by the time
  weekly JSONs are being processed. The snapshot value is useful as a baseline marker
  ("where did this athlete start?") but UNRELIABLE as a current-state indicator.
  Renamed to "baseline_season_phase" to prevent confusion with a future live field.
  CF fallback: "not available — core foundation intake"

  *** PRE-LAUNCH BLOCKER: LIVE SEASON PHASE ***
  A live current_season_phase field is required before app v1 launch.
  Proposed solution: Add a season_phase question to the weekly check-in submission
  (Off-Season / Pre-Season / In-Season / Post-Season). Stage 2 populates
  current_season_phase from the weekly input each week — always current, always accurate.
  This is a SEPARATE field from baseline_season_phase.
  Downstream consumers:
  - Stage 4 Season Phase Overlay dashboard metric (needs live data)
  - Stage 2 upcoming_context classification (could use live season context)
  - Stage 3 coaching tone (in-season vs off-season framing)
  This blocker must be resolved before v1 app launch. Design deferred to the weekly
  check-in input format specification (future task), but flagged here as a dependency.

COMMITMENT_LEVEL
  Source: Q28 (All-in / Ready to work / Willing to try / Not sure yet)
  Recommendation: NO ACTION
  Rationale: Static intake response with low downstream value. The athlete's actual
  commitment is revealed weekly through execution data (Win days, completion rates,
  engagement patterns). A "Willing to try" athlete who produces 7/7 Win weeks is
  functionally "All-in" regardless of what they checked at intake. Execution data
  supersedes self-reported commitment. Not worth the JSON payload.

GROUP B — PPD_SCORES
------------------------------------------------------------

ppd_primary_problem
  Source: Snapshot PPD_SCORES sub-block (deterministic bucket name, e.g., "Mistake Recovery Lag")
  Recommendation: PASS-THROUGH
  Rationale: This is the deterministic, machine-derived version of what
  baseline_primary_derailer says in prose. Having both gives Stage 2 a dual anchor:
  the prose sentence for narrative matching ("Loss of confidence following errors" maps
  to weekly behavioral evidence) and the bucket name for unambiguous categorical
  reference. Stage 3 can use the bucket name for coaching theme routing. Stage 2's
  Section 4C growth phase progression check continues to use the prose
  baseline_primary_derailer for adversity testing — the bucket name sits alongside
  but does NOT replace the prose anchor.
  CF fallback: "not available — core foundation intake"

ppd_top_3
  Source: Snapshot PPD_SCORES sub-block (3 ranked objects, each with: rank, bucket, score, trigger_context, coaching_implication)
  Recommendation: PASS-THROUGH
  Rationale: The coaching_implication field in each rank is directly actionable by
  Stage 3 for coaching theme selection. Example: rank_1 coaching_implication =
  "Reset reps + next-play language" tells Stage 3 exactly what coaching direction
  to take when the derailer activates. Currently Stage 3 must infer coaching themes
  from the prose baseline_primary_derailer — ppd_top_3 provides deterministic
  coaching direction. The full ranked list (not just rank 1) lets Stage 3 address
  secondary and tertiary patterns when relevant.
  CF fallback: "not available — core foundation intake"

ppd_all_scores
  Source: Snapshot PPD_SCORES sub-block (8 integer scores, one per bucket)
  Recommendation: NO ACTION
  Rationale: Explicitly stored for future longitudinal PPD Alignment Signal
  (comparing intake self-report PPD against execution-derived PPD). No current
  downstream consumer exists. Carrying 8 integer scores in every weekly JSON adds
  payload for zero current value. The comparison feature is a future enhancement —
  adding the field now would lock a schema structure before the consumption pattern
  is defined. When longitudinal PPD is designed, Stage 2 can start carrying it
  with zero disruption to existing fields.

ppd_tie_breaker_classification
  Source: Snapshot PPD_SCORES sub-block (Avoidance / Proving / Obligation / Approach, or dual classification)
  Recommendation: PASS-THROUGH
  Rationale: Informs Stage 3 coaching tone calibration. An Avoidance-framed athlete
  ("I don't want to fail") needs different coaching language than a Proving-framed
  athlete ("I need to show them"). Lightweight field (single enum or dual-enum value).
  Stage 3 can use it alongside emotional_intensity for nuanced tone selection.
  CF fallback: "not available — core foundation intake"

SECTION 3: PROPOSED SCHEMA CHANGES
============================================================

3.1 NEW JSON BLOCK: baseline_intake_profile

A new top-level block in the weekly Interpretation JSON, sitting alongside the
existing baseline_profile block. Separate block (not expanding baseline_profile)
because:

(a) The existing 5 baseline_profile fields are locked and consumed by established
    downstream rules. Modifying that block risks breaking existing Stage 3/4/5
    consumption patterns.
(b) The new fields are all intake-derived structured data with a fundamentally
    different character than the prose anchors in baseline_profile. Prose anchors
    are narrative sentences used for pattern matching. Intake profile fields are
    deterministic enums and structured objects used for routing and calibration.
(c) Clean CF fallback boundary: when input_source = "core_foundation", the entire
    baseline_intake_profile block populates with fallback values. Downstream stages
    can check input_source once and skip the block entirely.

Proposed schema:

"baseline_intake_profile": {
  "input_source": "app | core_foundation",
  "competitive_level": "Youth | Middle School | High School | College | Professional | not available — core foundation intake",
  "baseline_season_phase": "Off-Season | Pre-Season | In-Season | Post-Season | Year-Round Training | not available — core foundation intake",
  "ppd_primary_problem": "string (bucket name) | not available — core foundation intake",
  "ppd_tie_breaker_classification": "Avoidance | Proving | Obligation | Approach | [dual] | not available — core foundation intake",
  "ppd_top_3": [
    {
      "rank": "integer 1-3",
      "bucket": "string (bucket name)",
      "score": "integer",
      "trigger_context": "string | null",
      "coaching_implication": "string"
    }
  ],
  "abi_primary_emphasis": "array of 1-2 pillar names | ['Balanced'] | not available — core foundation intake",
  "adversity_response_pattern": "Push harder | Pull back and protect | Freeze and overthink | Reset and refocus | not available — core foundation intake"
}

  "competitive_level": "not available — core foundation intake",
  "baseline_season_phase": "not available — core foundation intake",

3.3 FIELD PLACEMENT RATIONALE

Why baseline_intake_profile and NOT expanding baseline_profile:

The existing baseline_profile block contains 5 prose anchors that are:
- Written by the Snapshot Builder agent (human-quality narrative sentences)
- Consumed by Stage 2 for alignment assessment and growth phase progression
- Consumed by Stage 3 for identity threading, micro-commitment grounding, and derailer address
- Consumed by Stage 4 for developmental context
- Consumed by Stage 5 for output verification

The new baseline_intake_profile block contains deterministic structured data that is:
- Computed by the Snapshot Builder from intake form responses (PPD scoring, ABI formula)
- Passed through Stage 2 unchanged (no classification or derivation)
- Consumed by Stage 3 for coaching calibration (competitive level, PPD coaching implications, ABI emphasis, adversity pattern, tie-breaker framing)
- Consumed by Stage 4 for contextual enrichment (competitive level, baseline season phase)
- Audited by Stage 5 for leakage compliance (PPD bucket names, ABI pillar names must not appear in coaching output)

These are fundamentally different data types with different provenance, different consumption patterns, and different compliance requirements. Separating them into distinct blocks preserves the integrity of the established baseline_profile contract while cleanly adding the new intake-derived data layer.

3.4 input_source FIELD NOTE

The baseline_intake_profile block includes its own input_source field. This mirrors
the execution_behavior_signals block which also has input_source. The duplication is
intentional: each block is self-describing, so downstream stages can check the block's
input_source without cross-referencing another block. The values will always be
identical (both read from the same snapshot), but each block carries its own for
independent consumption.

SECTION 5: DOWNSTREAM ROUTING IMPLICATIONS
============================================================

5.1 STAGE 3 — COACHING OUTPUT ENGINE

What Stage 3 can now do that it couldn't before:

(a) COMPETITIVE LEVEL CALIBRATION (NEW CAPABILITY)
    Stage 3 can read competitive_level and apply a formal Competitive Level
    Calibration Rule to adjust:
    - Vocabulary complexity (simpler for Youth/Middle School, sophisticated for College/Professional)
    - Directness level (more encouraging for younger athletes, more accountability-forward for older)
    - Accountability framing intensity (developmental for Youth, performance-standard for Professional)
    - Concept naming (age-appropriate language for mental performance concepts)
    This is a FORMAL RULE requiring a new section in Coaching Output Instructions.
    Flagged as Task 7 deliverable.

(b) PPD-DRIVEN COACHING THEMES (NEW CAPABILITY)
    Stage 3 can read ppd_top_3.coaching_implication for deterministic coaching
    direction when the derailer activates. Currently Stage 3 infers coaching themes
    from the prose baseline_primary_derailer. With ppd_top_3, the coaching implications
    are pre-computed and unambiguous.
    Example: rank_1 bucket = "Mistake Recovery Lag", coaching_implication = "Reset reps +
    next-play language" -> Stage 3 knows exactly what to reinforce.

(c) TIE-BREAKER FRAMING (NEW CAPABILITY)
    Stage 3 can read ppd_tie_breaker_classification to adjust coaching tone based on
    the athlete's pressure framing:
    - Avoidance-framed: "I don't want to fail" -> coaching reframes toward approach
    - Proving-framed: "I need to show them" -> coaching channels competitive energy productively
    - Obligation-framed: "I can't let them down" -> coaching separates performance from worth
    - Approach-framed: "I want to compete" -> coaching reinforces and deepens existing framing

(d) ABI EMPHASIS COACHING (NEW CAPABILITY)
    Stage 3 can read abi_primary_emphasis to weight coaching language toward the
    athlete's emphasis pillars. If emphasis = [Composure, Ownership], coaching
    naturally weights reset/composure language and accountability framing.

(e) ADVERSITY PATTERN CALIBRATION (NEW CAPABILITY)
    Stage 3 can read adversity_response_pattern to calibrate coaching during
    derailer activation weeks:
    - "Freeze and overthink" -> action-oriented coaching, break the freeze loop
    - "Push harder" -> modulation coaching, channel energy productively
    - "Pull back and protect" -> re-engagement coaching, safe re-entry framing
    - "Reset and refocus" -> reinforcement coaching, strengthen existing pattern

    Combined with emotional_intensity and competitive_level, this creates a
    three-dimensional coaching calibration matrix.

5.2 STAGE 4 — COACH INSIGHTS ENGINE

What Stage 4 can now do:

(a) COMPETITIVE LEVEL CONTEXT
    Stage 4 can include competitive level in the Weekly Performance Insight for
    contextual framing. Compliance-safe — competitive level is public information.
    Not a routing change but enriches institutional output.

(b) BASELINE SEASON PHASE REFERENCE
    Stage 4 can reference baseline_season_phase as a starting point. However, the
    Season Phase Overlay dashboard metric requires LIVE season phase data (see
    pre-launch blocker in Section 2). baseline_season_phase alone is insufficient
    for this metric.

(c) PPD/ABI FIELDS — INFORM-ONLY (COMPLIANCE BOUNDARY)
    PPD bucket names, ABI pillar names, and all intake-derived structured data
    follow the same inform-only pattern as execution signals. Stage 4 may use
    these fields internally to sharpen compliance-safe output, but:
    - PPD bucket names NEVER appear in institutional output
    - ABI pillar names/scores NEVER appear in institutional output
    - Adversity response pattern enum values NEVER appear in institutional output
    - Coaching implications are Stage 3 content, not Stage 4 content
    Stage 4 compliance framework already prohibits surfacing internal pipeline
    data to coaching staff. The new fields fall under the existing prohibition.

5.3 STAGE 5 — EDITORIAL AUDIT

What Stage 5 needs to audit:

(a) PPD/ABI LEAKAGE CHECK (NEW AUDIT CONCERN)
    Stage 5 must verify that PPD bucket names (e.g., "Mistake Recovery Lag",
    "Overload Fatigue") and ABI pillar names (as scoring labels, not as natural
    English words) do not appear in coaching output. This is analogous to the
    existing Criterion 13 (Execution Signal Leakage) — internal pipeline
    terminology must not leak into athlete-facing content.

    IMPORTANT: Words like "ownership," "composure," "focus," and "structure" are
    legitimate coaching vocabulary. The audit concern is their use AS PILLAR SCORE
    LABELS (e.g., "your Ownership score is low") not as natural coaching language
    (e.g., "take ownership of your preparation"). Same reasonable-reader test as
    Criterion 13.

    SCOPE DECISION: This concern can likely be folded into Criterion 13's existing
    framework (Category E expansion) rather than requiring a new criterion. Task 7
    will make the final determination.

(b) COMPETITIVE LEVEL CONSISTENCY CHECK
    Stage 5 should verify that coaching tone is consistent with the competitive_level
    field. A coaching message using sophisticated performance-psychology vocabulary
    for a Middle School athlete would be an audit flag. This is a new audit dimension
    that doesn't exist today.

    SCOPE DECISION: Whether this becomes a formal audit criterion or a quality note
    depends on how rigorous the Competitive Level Calibration Rule is in Stage 3.
    Task 7 will determine.

6.3 WHY THE BOUNDARY IS HERE

The boundary separates PASS-THROUGH (immediate) from CONSUMPTION RULES (downstream):

- Stage 2's job is to get the data into the JSON so downstream stages can use it.
  That's pass-through — low risk, well-understood pattern, already proven with
  execution_behavior_signals.

- How Stage 3 uses competitive_level for tone calibration, how Stage 4 handles
  PPD/ABI compliance, and how Stage 5 audits for leakage are all DOWNSTREAM
  CONSUMPTION decisions. Those belong in Task 7 (Stages 3-5 impact assessment),
  not in the Stage 2 assessment.

- The live season phase, longitudinal PPD, and Parent Output features are all
  FUTURE CAPABILITIES that require their own design cycles. Including them now
  would bloat the implementation scope and risk delaying the core integration.
